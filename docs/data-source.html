stack Terraform provider

The stack provider lets a Nuon install-stacks Terraform module (e.g. nuonco/install-stacks/gcp) talk to the Nuon control plane directly, instead of exchanging configuration and run status through generated tfvars and hand-rolled HTTP calls. It exposes two surfaces:

  • stack_config data source — reads the stack's rendered configuration (runner, permissions, roles, inputs, secrets) at plan time. Read-only.
  • stack_phone_home resource — reports the run result back to the control plane. The resource lifecycle drives the phone-home request_type (Create/Update/Delete).

Both are keyed by the per-stack-version phone_home_id, which acts as the secret — no Nuon API token is needed on the apply host.

Requiring the provider

terraform {
  required_providers {
    stack = {
      source = "nuonco/stack"
    }
  }
}

provider "stack" {
  # Optional; base URL up to but excluding /v1. Defaults to https://api.nuon.co.
  api_url = "https://api.nuon.co"
}

Architecture

Apply host / runner Nuon control plane Target cloud account (BYOC)

Walkthrough

  1. Read config at plan time. During terraform plan/apply, the data.stack_config block builds {api_url}/v1/stack-runs/{phone_home_id} and the SDK's FetchConfig issues a GET /config to the control plane. Read-only: no run is created and no status changes. ctl-api returns the rendered Config — install/org/app IDs, runner details, the phone-home URL, install inputs, secrets, and the per-operation permissions and roles. Because it resolves at plan time, values feed count/for_each directly.
  2. Render resources from the config. The install-stacks module references data.stack_config.this.* (typically via locals) in place of the variables it used to receive as tfvars, and terraform apply creates the install-stack resources — network, IAM service accounts/roles, secret entries, and runner compute — using the apply host's ambient cloud credentials. The GCP project/region are modeled as customer install inputs, so they arrive in install_inputs alongside the rest.
  3. Phone-home through the provider. The stack_phone_home resource depends on the module's outputs, so it fires after apply and POSTs the run result to .../phone-home/{phone_home_id} — replacing the module's hand-rolled HTTP call. The resource lifecycle drives request_type: Create on first apply, Update when outputs change, Delete on destroy.

stack_config data source

Reads an install stack's rendered configuration from the control plane — runner details, permissions, and roles — instead of receiving them as generated tfvars. The generated tfvars shrinks to just api_url, phone_home_id, and the customer-facing inputs; secrets are injected as a sensitive variable rather than written to disk. The data source provisions nothing: it is a read-only fetch keyed by phone_home_id.

Example usage

# install-stacks/gcp — the module reads its own config from the API.
# Only the customer-supplied inputs remain as variables.
variable "api_url"       { type = string, default = "https://api.nuon.co" }
variable "phone_home_id" { type = string }
variable "install_inputs" { type = map(string), default = {} }
variable "secrets"        { type = map(string), default = {}, sensitive = true }

provider "stack" {
  api_url = var.api_url
}

data "stack_config" "this" {
  phone_home_id = var.phone_home_id
}

locals {
  # stack config — used to arrive as ~20 tfvars, now read from the API.
  nuon_install_id = data.stack_config.this.install_id
  runner_id       = data.stack_config.this.runner_id
  runner_api_url  = data.stack_config.this.runner_api_url
  phone_home_url  = data.stack_config.this.phone_home_url

  runner_api_token       = data.stack_config.this.gcp.runner_api_token
  runner_init_script_url = data.stack_config.this.gcp.runner_init_script_url

  provision_permissions = data.stack_config.this.gcp.provision_permissions
  custom_roles          = data.stack_config.this.gcp.custom_roles

  # customer-facing values still come from the (slim) tfvars, including
  # the GCP project/region, which are modeled as customer install inputs.
  install_inputs = var.install_inputs
  secrets        = var.secrets
  gcp_project_id = var.install_inputs["gcp_project_id"]
  gcp_region     = var.install_inputs["gcp_region"]
}

resource "google_service_account" "provision" {
  project    = local.gcp_project_id
  account_id = "${local.nuon_install_id}-provision"
}

resource "google_project_iam_custom_role" "provision" {
  count       = length(local.provision_permissions) > 0 ? 1 : 0
  project     = local.gcp_project_id
  role_id     = "${local.nuon_install_id}_provision"
  permissions = local.provision_permissions
}
# nuon.auto.tfvars — the entire generated file. Everything else
# (runner, permissions, roles, phone_home_url, …) comes from the
# data source, so it is no longer rendered here.
api_url       = "https://api.nuon.co"
phone_home_id = "ph_abc123"

install_inputs = {
  gcp_project_id = "my-proj"
  gcp_region     = "us-central1"
  domain         = "example.com"
}

# secrets are NOT written to the tfvars file — they are injected as a
# sensitive variable so no secret value is persisted to disk:
#   export TF_VAR_secrets='{"db_password":"..."}'

stack_phone_home resource

Reports the result of an install-stack run back to the control plane through the provider, instead of the module constructing and sending the phone-home HTTP request itself. The resource lifecycle drives the phone-home request_type: Create on first apply, Update when the reported outputs change, Delete on destroy. The reported payload is carried opaquely as a JSON string, so the module keeps ownership of its own payload shape.

Example usage

data "stack_config" "this" {
  phone_home_id = var.phone_home_id
}

# ... module "stack" provisions the install-stack resources ...

resource "stack_phone_home" "this" {
  install_id      = data.stack_config.this.install_id
  phone_home_id   = var.phone_home_id
  phone_home_type = "gcp"

  # The module already computes these outputs; the provider POSTs them.
  # Keys match the GCPStackOutputs contract that deploy templates read back.
  # request_type + phone_home_type are injected by the provider.
  payload = jsonencode({
    project_id                   = var.gcp_project_id
    region                       = var.gcp_region
    runner_service_account_email = module.stack.runner_service_account_email
    break_glass_sa_emails        = module.stack.break_glass_sa_emails
    custom_sa_emails             = module.stack.custom_sa_emails
    install_inputs               = data.stack_config.this.install_inputs
  })
}

Lifecycle → request_type

Terraform lifecyclerequest_typeControl-plane effect
Create (first apply)Createrecords a run, marks the stack version active, enqueues the stackrun signal
Update (payload changed)Updatesame, recorded as an update
Delete (destroy)Deleteshort-circuits to 200 — no server-side processing; sent best-effort so destroy never wedges

Because the reported outputs (service-account emails, subnet names, …) are computed by the module's real resources, they are known only after apply — so the resource naturally re-reports whenever the infrastructure it depends on changes. ctl-api detects the cloud from the output keys (e.g. runner_service_account_email marks a GCP payload), so the key names are part of the contract, not just cosmetic.
